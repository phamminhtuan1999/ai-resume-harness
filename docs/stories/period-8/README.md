# Period 8 — AI Assistant Intelligence Overhaul

## Goal

Turn ApplyWise's existing analysis features from **deterministic placeholder
logic** into a real **AI job-hunting assistant**. The match analyzer, missing
skills, resume suggestions, resume draft, roadmap, and interview prep already
ship (US-007–US-011) but are generated by keyword matching and templated strings
in `apps/web/src/lib/*.mjs`. Every user-facing description, recommendation, and
decision must instead be generated by a model from the user's actual profile,
resume, job requirements, and saved analysis — with evidence, persistence,
regenerate, and graceful failure.

Source brief: `applywise_ai_assistant_update_tasks.md` (repo root). This period
makes the behavior already described in `docs/product/ai-workflows.md` real.

## Direction (accepted decisions)

Set by `docs/decisions/0012-ai-workflow-standards.md`:

- **AI runs in the FastAPI backend** (`apps/api`), using Pydantic schemas and
  the existing Gemini client pattern (`job_extractor.py`,
  `candidate_profile_extractor.py`). The web app calls the API; it no longer
  generates analysis inline.
- **Match-centric routing stays.** Reuse `/matches/:id/*`. The doc's
  `/jobs/:id/*` paths map onto the existing `matches` entity (resume × job).
  The Job Assistant Insight Card and AI Workflow Panel render on match/job
  detail over the active match.
- **Real Gemini is primary; the existing deterministic generators become the
  typed fallback** when no key is configured or generation fails. Working
  behavior never regresses.
- **Every AI feature follows one standard flow** (Feature 12): create an
  `ai_workflow_runs` row → run → validate with Pydantic → persist domain result
  + output snapshot → write an activity event → return. No raw resume/JD text in
  production logs.

## Sequencing

**Foundation first.** US-027 establishes the provider abstraction,
`ai_workflow_runs` + `activity_feed` tables, the `BaseAIWorkflow` flow, error
handling, and observability. Nothing else is built until US-027's pattern is
reviewable, because US-028–US-038 all reuse it.

Then by epic (per `applywise_ai_assistant_update_tasks.md` Appendix A):

1. **Epic 1 — AI Match Intelligence:** US-028, US-029, US-030.
2. **Epic 2 — AI Application Materials:** US-031, US-032, US-033.
3. **Epic 3 — AI Career Improvement:** US-034, US-035.
4. **Epic 4 — AI Assistant Experience:** US-036, US-037, US-038.

All twelve stories (US-027–US-038) now have full packets — high-risk folders for
US-027/US-028/US-031/US-033, flat files otherwise — plus per-story dev-flow docs
under `flows/`.

**Epics 1 and 2 are complete:** US-027–US-033 are all `implemented`; migrations
`0010`–`0014` are applied and REST-reachable. Each AI feature runs Gemini +
deterministic fallback through the US-027 foundation, with unit tests and a live
unauthenticated endpoint smoke (all wired + Clerk-enforced). Only authenticated
browser E2E remains as verification. US-034–US-038 (Epics 3–4) are planned and
should ship in epic order so each reuses the analysis the prior epic produced.

## Validation Shape

Per story: backend unit tests with deterministic provider fixtures (no live
model calls in unit tests) covering schema validation, retry, fallback, and
no-invention behavior; integration tests for the API endpoint + persistence +
workflow-run + activity rows; web E2E for the generate → view → regenerate flow.

## Out of Scope

- PDF/DOCX export, resume version diffing, model selector UI (still post-MVP per
  `docs/product/mvp-scope.md`).
- Browserbase/Stagehand, LangGraph orchestration (Feature 12 names these as
  post-MVP; the MVP uses a simple sequential backend workflow).
- Replacing Clerk auth, billing, or the design system.

## Exit Criteria

Every feature in the brief generates real AI output (or a schema-identical
fallback), validates with Pydantic, persists the result + a workflow run +
an activity event, shows personalized assistant text with evidence, supports
regenerate/retry, and handles failure with a friendly error — with zero raw
resume/JD text in production logs.
