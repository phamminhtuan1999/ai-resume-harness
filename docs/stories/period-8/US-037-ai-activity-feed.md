## Product Contract

Every `activity_feed` row written by `BaseAIWorkflow` (US-027) gains an
AI-generated `title`, `assistant_description`, and `importance` level produced
inline by a new `ActivityDescriptionHelper`. The dashboard surfaces a *Recent
Activity* section that renders up to 20 items newest-first, each with its
assistant description, importance badge, relative timestamp, and a related job
link when available. A standalone regenerate endpoint (`POST
/api/activities/{activityId}/generate-description`) lets users refresh a single
item. If AI generation fails at any level, deterministic fallback text is used
and the activity is still saved — no event is ever dropped.

This is Feature 10 of `applywise_ai_assistant_update_tasks.md`. It depends on
US-027 (foundation) — the `activity_feed` table and `BaseAIWorkflow` are already
implemented; this story enriches the rows they produce. It calls the external
provider through the boundary established in
`docs/decisions/0012-ai-workflow-standards.md` and is therefore a bounded,
normal-lane reuse rather than a new provider decision.

## Acceptance Criteria

- Given a workflow event occurs, when an activity is created, then the
  `activity_feed` row includes a non-null `title` and `assistant_description`
  generated by `ActivityDescriptionHelper`.
- Given the event carries a high hiring signal (match score ≥ 75, cover letter
  generated, interview prep completed), when the activity is saved, then
  `importance` is `"medium"` or `"high"`.
- Given AI description generation fails (invalid JSON, provider error, Pydantic
  failure), when the activity is written, then fallback text is used for
  `assistant_description` and the `activity_feed` row is still inserted — no
  event is dropped.
- Given `related_job_id` is set on an activity, when the user views the
  dashboard, then a link to `/jobs/{related_job_id}` is visible on that item.
- Given a user visits the dashboard, when `GET /api/activities` is called, then
  the response is ordered newest-first and includes `related_job` data when
  available.
- Given a user calls `POST /api/activities/{activityId}/generate-description` on
  an activity they own, when the request succeeds, then the DB row is updated and
  the endpoint returns the new description.
- Given a user calls the regenerate endpoint on an activity they do not own, then
  the response is `403 unauthorized`.
- Given Gemini is unavailable, when any workflow runs, then the deterministic
  fallback produces a valid `ActivityDescriptionOutput` and the activity is saved
  with `model_provider = deterministic` on the linked `ai_workflow_runs` row.
- Given the dashboard loads and there are no activities, then the empty-state
  message is shown (not an error).
- Given the regenerate call fails, when the user sees the result, then the
  original description remains visible and an error toast is shown (inline, never
  removes existing text).

## Design Notes

Full schemas, prompt text, mermaid diagrams, and the complete API response shape
are in `docs/stories/period-8/flows/US-037-ai-activity-feed-flow.md`.

- **Commands:** `ActivityDescriptionHelper.generate(event_context)` (new,
  `apps/api/app/services/ai/activity_description.py`). Called inside
  `BaseAIWorkflow` (`apps/api/app/services/ai/base_workflow.py`) before
  `insert_activity()`. Uses `GeminiProvider` / `DeterministicFallbackProvider`
  from `apps/api/app/services/ai/providers.py` — no new provider wiring.
  `workflow_type = activity_description` is written on standalone regenerate runs
  only.
- **Queries:** `GET /api/activities` (limit/offset, LEFT JOIN jobs for
  `related_job`).
- **API:** `GET /api/activities`, `POST
  /api/activities/{activityId}/generate-description`. New router
  `apps/api/app/routers/activities.py`; mounted in `apps/api/app/main.py`.
- **Tables:** No new table, no migration. Uses `activity_feed` created in
  US-027 migration `0010_period8_ai_workflow_foundation.sql`. The
  `assistant_description`, `importance`, and `title` columns exist and are
  nullable — this story fills them.
  Assumption: `0010_period8_ai_workflow_foundation.sql` is applied in all
  environments before US-037 ships.
- **Persistence helpers added to `apps/api/app/services/supabase_data.py`:**
  `list_activity_feed(user_id, limit, offset)` and
  `update_activity_description(activity_id, user_id, title, assistant_description, importance)`.
- **Settings:** No new keys. Existing `gemini_api_key`, `gemini_model`,
  `gemini_max_attempts`, `gemini_retry_base_delay_seconds` from
  `apps/api/app/settings.py` are sufficient.
- **Failure rule:** On any failure the helper returns a safe fallback —
  `activity_title` = human-readable form of `activity_type`; `assistant_description`
  = `"ApplyWise completed a {activity_type} workflow."`; `importance = "low"`.
  The calling workflow always proceeds to insert.
- **Domain rules:** Importance: `high` for direct hiring signals (score ≥ 75,
  cover letter generated, interview prep done); `medium` for useful-but-indirect
  signals; `low` for informational / system events. Max 2-sentence descriptions.
  AI speaks as ApplyWise ("ApplyWise scored…"). No invented data.
- **UI surfaces:** `apps/web/src/app/(app)/dashboard/page.tsx` — add *Recent
  Activity* section inside `<Suspense>`. New component
  `apps/web/src/components/dashboard/recent-activity.tsx`. Client convenience
  wrappers `fetchActivityFeed()` and `regenerateActivityDescription(id)` added to
  `apps/web/src/lib/ai-workflow-client.mjs`. Activity item layout: importance
  badge (high=amber, medium=blue, low=muted), title, `assistant_description`
  (2-line clamp), relative timestamp, related job link, hover regenerate icon
  button. States: loading skeleton (3 cards), empty message, success list, error
  + retry. Each item is an `<article>` with `aria-label`; badge uses
  `aria-label="Importance: {level}"`.
  Assumption: `BaseAIWorkflow._build_event_context()` has a default
  implementation; subclasses may override to pass richer `related_analysis` data.
